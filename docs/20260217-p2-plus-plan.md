## Phase 1: TYPE-1 — Zod Config Validation

**Files:** `src/config/loader.ts`, `src/config/schema.ts`, `src/config/validate.ts`, `package.json`
**Problem:** Config loader uses `as unknown as` double-casting (2 instances). No runtime type safety.

**Changes:**
1. Add `zod` dependency: `bun add zod`
2. In `src/config/schema.ts`, add a Zod schema (`NgentConfigSchema`) that mirrors the `NgentConfig` type. Use `z.infer<typeof NgentConfigSchema>` to derive the type (or keep manual type + validate against schema).
3. In `src/config/loader.ts`, replace the `deepMerge` + `as unknown as` pattern:
   - Merge raw objects (keep deepMerge for raw `Record<string, unknown>`)
   - Parse merged result through `NgentConfigSchema.safeParse()`
   - On failure, throw with Zod's formatted error messages
   - Remove the separate `validateConfig()` call (Zod handles it)
4. Remove or simplify `src/config/validate.ts` — Zod replaces manual validation.
5. Update tests in `test/config.test.ts` to test Zod validation errors.

**Run:** `bun test && bun run typecheck`
**Commit:** `refactor(config): replace manual validation with Zod schema parsing`

## Phase 2: BUG-4 — Batch-Wide Escalation Option

**Files:** `src/config/schema.ts`, `src/execution/runner.ts`
**Problem:** When a batch fails, only the first story escalates. Remaining retry at same tier — wasteful.

**Changes:**
1. In `src/config/schema.ts`, add to escalation config:
   ```ts
   escalateEntireBatch?: boolean; // default: true
   ```
   Add to DEFAULT_CONFIG and Zod schema.
2. In `src/execution/runner.ts`, in the batch failure handler:
   - If `config.autoMode.escalation.escalateEntireBatch` is true (default), escalate ALL stories in the failed batch to the next tier, not just the first.
   - If false, keep current behavior (only first story escalates).
3. Add tests for both modes.

**Run:** `bun test && bun run typecheck`
**Commit:** `fix(execution): add batch-wide escalation option (default: true)`
