# SPEC: Canonical Rules Store

> **Status:** Draft. Independent of the context engine and of fallback — shippable on its own. Eliminates per-agent rules-file drift (`CLAUDE.md`, `AGENTS.md`, `GEMINI.md`) by making a neutral store the single source of truth.

## Summary

Move project rules and conventions from `CLAUDE.md` + `.claude/rules/` into a neutral canonical store at `.nax/rules/` that is **read by nax regardless of which agent runs the session**. Per-agent rules files (`CLAUDE.md`, `AGENTS.md`, `GEMINI.md`, etc.) become one-way exports generated by `nax rules export --agent=<id>` for third-party tools running outside nax. The engine never reads them at runtime.

This removes three classes of problem:

1. **Drift between agent-specific rule files.** Today a new convention added to `CLAUDE.md` is invisible to Codex or Gemini unless manually duplicated.
2. **Agent-specific phrasing creeping into project rules.** `CLAUDE.md` references Claude Code tools ("Use the Grep tool") that mean nothing to other agents.
3. **The "which file is the source of truth?" question** when multiple agents are in play or when fallback switches agents mid-story.

## Motivation

Currently:

- `CLAUDE.md` is the root rules file, loaded into every Claude session.
- `.claude/rules/*.md` provides detailed per-topic rules (project conventions, testing, error handling, etc.).
- These files are authored for Claude and contain Claude-specific phrasing (`<system-reminder>`, tool references, Claude-style directives).
- When nax runs Codex or another agent, those files are either ignored (the agent reads `AGENTS.md` instead) or loaded inappropriately (phrasing confuses the agent).

As nax expands to support multiple agents (driven by the fallback spec and routing flexibility), the "one rules file per agent" model scales poorly. Adding Gemini means maintaining a third file. Every rule change means three edits. Drift is inevitable.

The fix is architectural: nax maintains one neutral store and injects it directly into every agent's prompt, bypassing the agent's native rules-file convention. Agents' native files become optional exports for humans invoking those agents directly, outside nax.

## Non-goals

- Not a rules authoring framework. Operators still write Markdown. We add a lint step to catch agent-specific phrasing, not a DSL.
- Not a translation engine. We don't rewrite Claude-specific sentences into Codex-specific ones. Content is authored neutral from the start.
- Not a migration shortcut. Converting existing `CLAUDE.md` content to neutral form requires human judgment; we provide tooling to help but not to do it unattended.
- Not a dependency of the context engine or fallback specs. This spec stands alone. It can ship with or without either.

## Design

### Canonical store layout

```
.nax/rules/
├── index.md                       # optional top-level overview / pointers
├── coding-style.md
├── project-conventions.md
├── forbidden-patterns.md
├── testing.md
├── error-handling.md
├── config-patterns.md
├── git-workflow.md
├── security.md
├── performance.md
├── agents.md                      # rule-usage guidance (which rules apply when)
└── patterns.md
```

File names and topic splits are up to the project. The engine treats `.nax/rules/*.md` as a flat list; nested directories are supported but not required.

### Neutrality requirements

Rules content must be **agent-neutral**. The neutrality linter rejects files containing any of:

| Banned | Reason | Replace with |
|:-------|:-------|:-------------|
| `CLAUDE.md`, `AGENTS.md`, `GEMINI.md` references | Agent-specific file names | "the project rules store" or inline content |
| `.claude/`, `.codex/`, `.gemini/` directory paths | Agent-specific | "the rules store" |
| `<system-reminder>` tags, `<ide_diagnostics>`, other Claude-CLI-specific XML | Injected by Claude Code; not content | Remove |
| "the Grep tool", "the Read tool", "the Bash tool" | Claude-CLI tool names | "search files for pattern", "read a file", "run a shell command" |
| "Use X instead of Y" where X is a Claude-specific tool | Same | neutral capability phrasing |
| `IMPORTANT:`, emoji, all-caps shouting | Rhetorical amplification that agents differ on following | Plain statements |
| "the assistant" / "you as the assistant" | Agent-persona phrasing | "the agent" or second-person |
| Agent-specific prompt directives (`Think hard`, `ultrathink`, etc.) | Claude-specific | Remove or neutralize |

Linter is pragmatic, not perfect. Its job is to catch obvious leaks, not to guarantee universal portability. Operators override individual lints per file with a `<!-- nax-rules-allow: <rule-id> -->` marker when they have a good reason.

### Loader

`src/context/rules/canonical-loader.ts`:

```typescript
export interface CanonicalRule {
  id: string;                 // file stem, e.g. "coding-style"
  path: string;               // .nax/rules/coding-style.md
  content: string;            // verbatim Markdown, lint-validated
  tokens: number;
  appliesTo?: string[];       // optional glob(s) for file/context scoping
}

export async function loadCanonicalRules(workdir: string): Promise<CanonicalRule[]>;
```

Behavior:

1. Read `.nax/rules/*.md` (flat glob, depth-1 nesting allowed).
2. For each file, run the neutrality linter.
3. On lint failure, throw a `NaxError` with `code: "RULES_NEUTRALITY_VIOLATION"` listing offending lines. Loading fails loud — silently accepting a Claude-specific rule defeats the point.
4. Parse optional frontmatter for `appliesTo` (e.g., `appliesTo: ["src/agents/**"]` scopes a rule to work in the agents directory).
5. Return the sorted, validated list.

Results are cached per-workdir per-run; the store is small (tens of KB typical).

### Injection

Rules are injected into every agent's prompt by a new stage-0 provider or by existing prompt builders (pick one — see §Integration below). The injected section looks like:

```
## Project rules

<inline content of .nax/rules/index.md if present>

<for each rule file:>
### <rule.id>

<rule.content>

---
```

Total rules budget is capped at `config.rules.budgetTokens` (default 8192). If the total exceeds the budget, the loader emits a warning and truncates tail-biased (keep files with higher priority; see below).

**Priority ordering:**

- Frontmatter `priority: <n>` (default 100). Lower = earlier = less likely truncated.
- Absent frontmatter → alphabetical by file name.

### Integration — two options

**Option A (minimal):** Replace the existing `CLAUDE.md` + rules loading path in prompt builders with the canonical loader. Every builder that currently reads those files now reads `.nax/rules/`. One-line change per builder.

**Option B (with context engine):** Canonical rules become a provider (`StaticRulesProvider`) in the context engine. Builders consume the engine's bundle instead of reading rules directly.

Option B is cleaner but depends on the context engine existing. **This spec ships with Option A** to avoid cross-dependence. When the context engine lands (if it does — gated on Phase 0 validation), it subsumes Option A's code into a provider with no operator-visible change.

### Export — `nax rules export`

For humans running `claude-code`, `codex`, or `gemini-cli` *directly* in the same repo (outside nax), generate per-agent shim files on demand:

```
nax rules export --agent=claude [--out=CLAUDE.md]
nax rules export --agent=codex  [--out=AGENTS.md]
nax rules export --agent=gemini [--out=GEMINI.md]
```

Behavior:

- Concatenates `.nax/rules/*.md` into the target file.
- Adds a header: `# Project rules (auto-generated from .nax/rules/ by nax rules export — do not edit)`.
- Optionally wraps in agent-native framing (Claude: plain Markdown; Codex: XML sections if agent profile declares it).
- **One-way.** The command overwrites the target. Any manual edits are lost.
- No per-agent content rewriting — the content is already neutral; the export is effectively a copy with a header.

Generation is optional. Projects that don't run agents outside nax can ignore it. The `.nax/rules/` store is all nax needs.

### Migration path

For projects with existing `CLAUDE.md` + `.claude/rules/`:

#### `nax rules migrate`

```
nax rules migrate [--from=CLAUDE.md] [--from=.claude/rules/] [--to=.nax/rules/] [--dry-run]
```

Behavior:

1. Read the source files.
2. Run a **neutralization pass**:
   - Strip `<system-reminder>` tags and contents.
   - Rewrite `the Grep tool` → `grep` (neutral CLI term), `the Read tool` → `read a file`, etc., via a static substitution table.
   - Flag lines the substitution table doesn't know how to handle; these are emitted as `TODO: neutralize —` comments inline for the operator to resolve.
   - Strip emoji, `IMPORTANT:` prefixes, all-caps exhortations.
   - Preserve headings and structure; rewrite file references (`.claude/rules/X.md` → `.nax/rules/X.md`).
3. Write the draft to `.nax/rules/` (or show diff in `--dry-run`).
4. Run the neutrality linter against the draft. Any remaining violations are listed for manual fix.
5. **The operator reviews and commits.** The tool produces a draft; it does not auto-commit.

**Pre-cutover coexistence:** during migration, allow the engine to fall back to the legacy files when `.nax/rules/` is absent:

```typescript
// In loader:
if (!(await directoryExists(`${workdir}/.nax/rules/`))) {
  if (config.rules?.allowLegacyClaudeMd) {
    logger.warn("rules", "Using legacy CLAUDE.md fallback — migrate to .nax/rules/", { workdir });
    return loadLegacyClaudeRules(workdir);
  }
  logger.warn("rules", ".nax/rules/ not present; no rules loaded", { workdir });
  return [];
}
```

`allowLegacyClaudeMd` defaults to `true` for one minor version after this spec ships, then the default flips to `false`. Operators who haven't migrated by then see the warning and decide.

### Scoping — `appliesTo`

Optional frontmatter:

```markdown
---
appliesTo:
  - "src/agents/**"
  - "test/integration/agents/**"
priority: 50
---

# Adapter wiring rules

...
```

When the current story touches files matching `appliesTo`, the rule is injected. Otherwise it's omitted. This replaces the existing `.claude/rules/` path-scoping mechanism (which scopes via directory placement — `.claude/rules/src/agents/...`). The new mechanism is explicit and agent-agnostic.

If no `appliesTo` is present, the rule is always injected.

### Prompt placement

Rules appear in the agent prompt **before** the story description, matching current behavior. Section header `## Project rules`, one level of indentation per rule file.

- `docs/authoring-neutral-rules.md` — style guide for writing neutral rules.

### Modified

- `src/prompts/builders/*.ts` — any builder currently reading `CLAUDE.md` or `.claude/rules/` switches to the canonical loader.
- `src/context/injector.ts` — if it currently loads rules, route through the canonical loader.
- Existing `CLAUDE.md` — during migration, regenerated from `.nax/rules/` via `nax rules export --agent=claude`. After the migration window, the repo can either keep the regenerated shim or delete it; both work.
- `.claude/rules/*.md` — moved content goes to `.nax/rules/`. Original files can be deleted after migration; the engine stops reading them.
- `.gitignore` — no change; `.nax/rules/` is committed (it's project config).

### Unchanged

- `src/constitution/`, `src/context/` non-rules paths, review pipeline, verification, TDD stages. Rules are just more context; how they're loaded is an implementation detail to downstream code.

## Risks

### Authoring burden is underestimated

Converting `CLAUDE.md` to neutral form requires judgment, especially for passages that describe workflows ("Use the TodoWrite tool to track multi-step work"). **Mitigations:**

- `nax rules migrate` handles the mechanical substitutions and flags the rest with `TODO: neutralize` inline comments.
- `docs/authoring-neutral-rules.md` provides worked examples for the tricky cases.
- Dogfooding on the nax repo itself establishes the template and catches edge cases before external projects hit them.
- Linter exceptions via `<!-- nax-rules-allow: <rule-id> -->` let operators bypass individual lint rules for legitimate cases, rather than being forced to rewrite perfect content to satisfy the linter.

### Agents running outside nax miss updates to rules

A developer who runs `claude-code` directly in the repo sees `CLAUDE.md`, but `CLAUDE.md` is now a generated shim that only updates when someone runs `nax rules export`. If the shim goes stale, the human-facing agent diverges from the nax-facing agent. **Mitigations:**

- Pre-commit hook or Husky integration that regenerates shims when `.nax/rules/*.md` changes.
- Check-in the shim alongside source changes so git history shows the drift.
- Document the workflow explicitly: *edit `.nax/rules/`, then run `nax rules export` before committing.*

### Existing tools assume `CLAUDE.md` at the repo root

Some tooling or documentation in the repo may expect `CLAUDE.md` to be the authoritative source. **Mitigations:**

- `nax rules export --agent=claude` keeps `CLAUDE.md` present and correct; external tools are not broken.
- The file just becomes generated; anyone who grew a habit of editing it learns (via the generated-header comment) to edit the canonical source instead.
- Migration communication: a `NEWS` or `CHANGELOG` entry makes the source-of-truth change explicit.

### Neutrality linter false positives

A rule legitimately needs to discuss Claude-specific behavior (e.g., documenting a Claude-only bug workaround). The linter flags it as a violation. **Mitigations:**

- Per-file and per-line overrides via `<!-- nax-rules-allow: <pattern-id> -->`.
- Linter errors are actionable — they cite the offending line, the reason, and the override syntax.
- Over time, the linter's rule set is tuned based on real override usage; patterns that get overridden frequently either become allowed by default or get a better replacement recommendation.

### Legacy fallback hides migration debt

Projects running with `allowLegacyClaudeMd: true` (the default initially) may never migrate. The canonical store concept becomes aspirational. **Mitigations:**

- Deprecation warning every run: *"Using legacy `CLAUDE.md` — migrate via `nax rules migrate`."*
- Hard deadline: `allowLegacyClaudeMd` default flips to `false` after one minor version. A project that still hasn't migrated sees "no rules loaded" warnings — a stronger signal.
- Metrics log how many runs still use the legacy path; nax maintainers see aggregate adoption.

### Rule content growing to exceed budget

As projects accumulate rules, total tokens can exceed `budgetTokens`. Truncation drops rules the operator cares about. **Mitigations:**

- `priority` frontmatter lets operators mark which rules must survive truncation.
- `appliesTo` frontmatter scopes rules to relevant files, keeping most rules out of most prompts.
- Linter warning when total tokens approaches 75% of budget — encourages splitting / scoping before truncation hits.

## Open questions

1. **Frontmatter or plain Markdown?** Using YAML frontmatter adds a dependency (YAML parser) and a learning cost. Alternative: a special heading convention (e.g., `<!-- priority: 50 -->` HTML comments). Tentative: YAML frontmatter — it's standard, most Markdown tools understand it, the parser cost is trivial.

2. **Where does `appliesTo` glob evaluate?** Against the touched files in the current story, or against the working directory? Tentative: against `ctx.story.touchedFiles` if known, else against the current working set from the PRD.

3. **Do we import the existing `.claude/rules/README.md` organization?** That file organizes rules as common + language-specific (`common/`, `typescript/`, `python/`, etc.). The canonical store could mirror this. Tentative: yes, allow subdirectories for organization; the loader flattens.

4. **Priority of `index.md`.** If present, is it always first? Tentative: yes, hardcoded first; its priority defaults to 0.

5. **Shim generation hook.** Should `nax rules export` be a pre-commit hook by default, or opt-in? Auto-hooks are invasive; opt-in risks staleness. Tentative: opt-in, with clear documentation.

6. **Per-agent content variation.** Suppose a rule needs slightly different wording for different agents. Options: (a) inline conditional blocks (`{{ if agent == "claude" }}...{{ end }}`), (b) per-agent override files under `.nax/rules/agents/<id>/`, (c) don't support — neutral prose should cover all cases. Tentative: (c), revisit only if real cases arise.

7. **Linter as first-class vs optional.** Should loading succeed with lint warnings, or fail on any violation? Tentative: fail by default, with per-rule override markers. A strict default teaches the team the constraint; overrides handle real exceptions.

8. **Backwards compatibility for subagent rule files.** `.claude/rules/project-conventions.md` has `project-conventions.md` sibling elsewhere (e.g., under language-specific subdirs). Migration tool needs to handle this; tracked as an implementation detail.

## Acceptance criteria

1. **Loader reads `.nax/rules/`.** Given `.nax/rules/coding-style.md` with neutral content, the loader returns one `CanonicalRule` with the expected fields and token count.

2. **Loader respects flat + nested layout.** Both `.nax/rules/x.md` and `.nax/rules/sub/y.md` are loaded. Files at depth > 1 are ignored with a warning (prevents unbounded recursion).

3. **Frontmatter parsing.** A file with YAML frontmatter `priority: 50` and `appliesTo: ["src/**"]` produces a `CanonicalRule` with those fields correctly parsed. Malformed frontmatter fails loading with a clear error.

4. **Neutrality linter — banned patterns.** A rule containing `the Grep tool`, `<system-reminder>`, `CLAUDE.md`, `.claude/`, or emoji is rejected with a `NaxError` citing the file and line.

5. **Neutrality linter — overrides.** A rule with `<!-- nax-rules-allow: claude-reference -->` on the line containing `CLAUDE.md` is accepted. The override applies only to that line.

6. **Budget truncation.** With total rule tokens exceeding `budgetTokens`, lower-priority rules are truncated (tail-biased by priority then alphabetical); a warning is logged. Higher-priority rules always survive.

7. **Scoping by `appliesTo`.** A rule with `appliesTo: ["src/agents/**"]` is included for a story touching `src/agents/acp/unified.ts` and excluded for a story touching `src/review/semantic.ts`.

8. **Prompt integration.** The canonical rules appear under a `## Project rules` section in the agent prompt, before the story description. All active rules (after `appliesTo` filtering) are concatenated.

9. **Legacy fallback — enabled.** With no `.nax/rules/` present and `allowLegacyClaudeMd: true`, the loader reads `CLAUDE.md` + `.claude/rules/*.md`, bypassing the neutrality linter, and emits a deprecation warning.

10. **Legacy fallback — disabled.** With no `.nax/rules/` present and `allowLegacyClaudeMd: false`, the loader returns zero rules and emits a warning. The pipeline continues (rules are not strictly required to run).

11. **Export — claude.** `nax rules export --agent=claude` writes a `CLAUDE.md` containing the concatenated `.nax/rules/` content prefixed with a generated-file header. Running a Claude session outside nax against that `CLAUDE.md` surfaces the same rules.

12. **Export — codex.** `nax rules export --agent=codex` writes `AGENTS.md` with equivalent content in the agent's preferred framing (per the agent profile registry, if present; otherwise identical to the Claude export).

13. **Export — overwrite.** Running export twice produces identical files. Manual edits between runs are lost (documented as expected).

14. **Migrate — dry run.** `nax rules migrate --dry-run` prints the neutralization diff without writing. The output is deterministic for the same input.

15. **Migrate — write.** `nax rules migrate` without `--dry-run` writes the draft `.nax/rules/` directory. The linter is run on the result; remaining violations are listed as `TODO: neutralize` inline comments or as a final error message.

16. **Migrate — idempotent on neutral input.** Running `nax rules migrate` against a `.nax/rules/` store that's already neutral produces no changes.

17. **Config.** `context.rules.enabled: false` disables rules loading entirely; the prompt has no `## Project rules` section. `context.rules.budgetTokens: 4096` enforces the 4096-token ceiling.

18. **No impact on review/TDD pipelines.** With the canonical loader in place, existing tests for review, TDD, verification, and rectification continue to pass without modification (rules content is the same; only the load path changed).

19. **Dogfood milestone.** The nax project itself completes the migration: its own rules live in `.nax/rules/`, its `CLAUDE.md` is auto-generated by `nax rules export --agent=claude`, and a pre-commit hook (opt-in) ensures the shim stays current.

20. **Deprecation-path adherence.** After one minor version with `allowLegacyClaudeMd: true` as default, the default flips to `false`. The legacy loader and `CLAUDE.md` fallback path are removed entirely one minor version after that. No surprises; change is communicated via release notes.
