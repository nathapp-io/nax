# BUG-067: Fix stories and parallel batch execution use CLI adapter instead of ACP

**Severity:** Medium
**Component:** `src/execution/lifecycle/acceptance-loop.ts`, `src/execution/parallel-coordinator.ts`
**Found:** 2026-03-16
**Status:** Open

## Summary

When `config.agent.protocol = "acp"`, fix stories (generated by the acceptance retry loop) and
parallel batch stories execute using the legacy CLI adapter instead of the ACP adapter. They lose
ACP session management — no named sessions, no session resume on retry, no sidecar tracking.

## Root Cause

There are two `getAgent()` functions in `src/agents/registry.ts`:

1. **Module-level `getAgent()`** (line ~32) — always returns CLI adapter from `ALL_AGENTS.find()`
2. **Registry `getAgent()`** (inside `createAgentRegistry()`) — protocol-aware, returns ACP adapter when configured

The pipeline's `execution` stage uses `ctx.agentGetFn ?? _executionDeps.getAgent`. When `agentGetFn`
is missing from the `PipelineContext`, it falls back to `_executionDeps.getAgent` which imports the
module-level function — always CLI.

`iteration-runner.ts` correctly threads `agentGetFn: ctx.agentGetFn` into its `PipelineContext`.
The following call sites do NOT:

| File | Context | Impact |
|:-----|:--------|:-------|
| `acceptance-loop.ts:~135` | `fixContext` (fix story execution) | **High** — confirmed in production log |
| `parallel-coordinator.ts:~148` | `baseContext` (parallel batch) | **High** — all parallel stories affected |
| `sequential-executor.ts:~73` | `preRunCtx` (acceptance test setup) | Low — pre-run pipeline doesn't call agent |

## Evidence

From `nax-logs/acp-hello-auth-module/2026-03-16T07-03-29.jsonl`:

```
{"stage":"acp-adapter","message":"Starting run for claude",...,"storyId":"US-001"}  // regular stories: ACP ✓
{"stage":"pid-registry","message":"Registered PID 70635"}                            // US-FIX-001: CLI ✗
```

US-FIX-001 shows `pid-registry` logs (CLI pattern) with no `acp-adapter` stage logs.
Cost uses "duration-based" fallback — another CLI adapter indicator.

## Fix

Thread `agentGetFn` through each affected `PipelineContext`:

```typescript
// acceptance-loop.ts — fixContext
const fixContext: PipelineContext = {
  ...existing fields,
  agentGetFn: ctx.agentGetFn,  // ADD
};

// parallel-coordinator.ts — baseContext
const baseContext = {
  ...existing fields,
  agentGetFn: options.agentGetFn,  // ADD (also add to function signature)
};
```

## Notes

- `AcceptanceLoopContext` already has `agentGetFn?: AgentGetFn` in its interface — just not wired to `fixContext`
- `parallel-coordinator` function signature needs `agentGetFn` added as a param from `runParallelExecution`
- The `sequential-executor.ts:73` preRunCtx is low priority — its pipeline stages don't invoke agents
